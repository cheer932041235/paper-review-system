# Methodology · 审稿方法论

本文档讲清两档审稿背后的**判断逻辑与纪律**——工具只是载体，这些原则才是真正可迁移的东西。

---

## 一、数值交叉验证（皇冠上的明珠，最高优先级）

**核心**：全文每个数字在每处都必须一致；每个"算出来的数字"都必须真的等于它的算式。

**为什么排第一**：数字对不上是审稿人成本最低、杀伤力最大的击杀——它客观（对错一眼定）、它**全局摧毁信任**（"连这都能错，其它结论还能信吗"）、它最容易在多轮改稿中悄悄 drift。别的维度是"好不好"，数值一致是"可不可信"——可信是地基。

查这几类：

1. **同一数字多处一致**：摘要 ↔ 正文 ↔ 表 ↔ 图 ↔ caption。
2. **派生数字 = 算式（最危险）**：每个 "+X" / "Δ" / 倍数 / 占比手算回去。注意陷阱——"先 round 还是先减"能差一个尾数（`0.686−0.474=0.21` vs `0.69−0.47=0.22`），挑剔的审稿人会拿这质疑你。
3. **"看起来不一致"必须有交代**：要么证明一致（口径不同），要么解释差异（如"这一列是另一条管线的零噪点，故与基准表差 0.01"），**绝不沉默留着**。
4. **不同聚合口径会"对不上"但可能都对**：per-recording 配对均值 ≠ seed 均值之差——都对，但审稿人一定问，要主动说明。
5. **求和 / 公式验算**：分项和 = 100%；`F1 = 2PR/(P+R)`；`per-query cost = total/N`；κ 与 agreement% 关系合理。

**进阶 · 数据真实感**（数据被补充/美化时）：尾数分布要散（别全 .0/.5）、衰减别太对称、消融 drop 别成等差数列、改进幅度符合领域常识。真实数据有自然毛刺，编的数据太工整——这层专抓 AI 编造/过度美化。

**怎么做**：① 单一数据源（一个 `results.json`，正文/表/图全从它取——数字活在两处必 drift）；② grep 每个数字回溯到源；③ 每个派生数字手算；④ 不一致只有两个出口：证明一致 或 解释差异。

## 二、诊断看全局，治疗要精确（两阶段原则）

- **阶段一·全局诊断**：一次性截全部页、批量读完、建**完整问题清单**（页码 + 问题 + 根因）。先看全貌，因为问题常有关联。
- **阶段二·逐个治疗**：从最严重/最上游开始，**一次只改一处**，改完**立即重编译 + 定向截图验证**，确认无副作用再动下一个。

> 可迁移到任何复杂系统：诊断需要全局视野，治疗必须精确可控。

**LaTeX 浮动体血泪**：图/表位置**绝不能批量改**。一次性加多个 `\FloatBarrier` + 批量 `[t]→[b]` 会导致表文分离 + 大面积白页，比原问题更糟——浮动体是全局优化问题，改一处影响后面所有页。阶梯式最轻量优先：微调 float 参数 → 单独调那一个 `[htbp]` → 只在大章节边界加 `\FloatBarrier` → `[H]` 是最后手段。

## 三、为什么要"用眼睛看成品"

**源码对 ≠ 成品对**。一个数字在源文件里完全正确，编译出来可能图被裁、caption 跑版、整页空白、公式没渲染。文本审稿永远看不到排版错。所以必须把 PDF 转成逐页图片、用多模态"看"一遍——很多致命的视觉问题（如图编号顺序、跨页表格断裂）只有把图和正文摆一起看才发现。

---

## 四、多模型终审：用独立性对冲偏见

单个 LLM 审稿有系统偏差（偏爱熟悉写法、漏自己也会犯的错）。解法不是问一个模型 6 遍（放大同一偏见），而是换 6 个不同训练数据的模型家族。**不同来源的模型都指出同一问题 → 大概率是真的；只有一个提 → 可能是它的幻觉。**

**Devil's Advocate 是关键设计**：多模型默认会"互相客气"（都倾向 accept），专塞一个被指令去反驳的角色拉低天花板、暴露弱点。你自己写论文前也该养一个内心的 Devil：「最刻薄的审稿人会怎么打我？」

**聚合逻辑（务必分清两条线）**：
- **分数 → 取平均**：5 个审稿人各从 6 维打分，每维取算术平均（+ Min/Max 分歧区间）。
- **意见 → 取并集 + 去重 + 分诊**：Meta 把 5 人的 weakness 汇总、合并重复、triage。
- **Meta 不打分**，只综合裁决。

## 五、评分判读三层法

六维全 1–10 + Confidence 1–5。

**绝对校准**：9–10 顶尖（罕见）/ 8 强 / 7 扎实可录 / 6 刚过线 / 5 刚不过 / 4 不够格 / ≤3 明确拒。

判断分三层：
1. **绝对值**：先有尺子。
2. **按 venue 加权（最关键）**：顶会重 Originality + Significance；四区/工程刊重 Soundness + Clarity + Reproducibility。**同一份分，不同 venue 好坏可相反。**
3. **看「形状」而非总分**：高执行 + 中原创 = 干净的应用型论文（四区稳、顶会悬）；Min–Max 区间 = 分歧大小。

**口诀：总分定生死线，形状定论文性格，venue 定形状好不好。**

**Triage 四分类**（比分数更有用）：
- `MUST FIX` — 真方法学/表述缺陷，必改。
- `NICE TO HAVE` — 有改进空间非必需。
- `ALREADY ADDRESSED` — 论文已声明、审稿人重复提，一般不改。
- `REVIEWER ERROR` — 审稿人可能误判，解释清楚不盲从。

---

## 六、局限（别迷信）

LLM 审稿会编造问题、也会漏真错（所以要 6 模型交叉 + Devil + Meta 兜底）；同家族模型会趋同（刻意选不同厂商 + 双 API fallback）；regional 校准会放宽（别拿 regional 的高分当顶会）；换面板分数会小幅漂（盯共识方向和 must-fix，别盯 0.2 波动）；它替代不了真期刊人类终审——它是"投稿前自检的镜子"。

---

## 七、taste 总提炼

1. 两档制 = 成本纪律：高频免费自检兜底，关键节点付费终审。
2. 数值对不上是最廉价的击杀——投稿前把数值一致性当红线。
3. "看起来不一致"只有两个合法出口：证明一致，或解释差异。
4. 诊断看全局、治疗要精确、紧耦合系统绝不批量改。
5. 必须用眼睛看成品——源码对不等于 PDF 对。
6. 独立性对冲偏见：6 个不同厂商 > 一个模型问 6 遍；养一个内心的 Devil。
7. 看分数的形状和 venue，不看绝对总分。
8. 底层是**诚信**：数值交叉验证、数据真实感、不迷信分数——都是"我对我报告的每个数字、每个声明负责"。
